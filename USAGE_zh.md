## 1. Git Agent 是什么？

Git Agent 是一个**文件版本管理助手**。打个比方：

> 📖 就像您写报告时每次修改都会另存一个副本（"报告_v1.doc"、"报告_v2.doc"、"报告_终稿.doc"），Git Agent 自动帮您做这件事，而且做得更好——它不会产生一堆文件，而是安静地记录每一次修改，您可以随时查看历史、对比差异、甚至恢复到任意一个旧版本。

**核心好处：**

| 场景 | 没有 Git Agent | 有了 Git Agent |
|------|---------------|----------------|
| 文件改坏了想回退 | 找不到旧版了 😱 | 一句话恢复旧版本 ✅ |
| 想知道谁改了什么 | 只能挨个问同事 | 一句话查看修改记录 ✅ |
| 多人同时编辑冲突 | 文件互相覆盖 😱 | 自动检测并协助解决 ✅ |
| 想对比两次修改 | 手动打开两个文件对 | 一句话查看差异对比 ✅ |

## 2. 安装与启动

### 2.1 安装前提

- 您的电脑需要安装 **Go 语言环境**（1.24 或更高版本）
  - 检查方法：打开终端输入 `go version`，如果显示版本号则已安装
  - 如未安装，请访问 https://go.dev/dl/ 下载安装

### 2.2 下载项目

打开终端（macOS 在「启动台」搜索「终端」），输入：

```bash
git clone <项目地址> git-agent
cd git-agent
go mod tidy
```

### 2.3 启动方式

#### 方式一：本地模式（推荐新手先用这个）

不需要任何额外配置，直接运行：

```bash
go run main.go
# 或：make dev
```

#### 方式二：LLM 模式（更智能，需要 API Key）

如果您有大模型 API Key（如 OpenAI、DeepSeek 等），可以使用更智能的模式：

```bash
# 使用 OpenAI
go run main.go --api-key sk-你的密钥 --model gpt-4o

# 使用 DeepSeek（国产，更便宜）
go run main.go --api-key sk-你的密钥 --base-url https://api.deepseek.com/v1 --model deepseek-chat

# 使用 Azure OpenAI
go run main.go --api-key 你的密钥 --base-url https://你的名称.openai.azure.com/openai/deployments/你的模型 --model gpt-4o

# 使用本地 Ollama（免费，无需云端 API！）
go run main.go --api-key ollama --base-url http://localhost:11434/v1 --model qwen2.5:7b
```

> 💡 **小贴士**：也可以用环境变量代替命令行参数，这样不用每次都输入：
> ```bash
> export GIT_AGENT_API_KEY=sk-你的密钥
> export GIT_AGENT_BASE_URL=https://api.deepseek.com/v1
> export GIT_AGENT_MODEL=deepseek-chat
> go run main.go
> ```

#### 🦙 使用本地 Ollama（免费，无需联网）

如果您本地安装了 [Ollama](https://ollama.ai)，可以用本地大模型运行 Git Agent —— 完全免费且离线可用！

**第 1 步：安装 Ollama**

访问 https://ollama.ai 下载并安装 Ollama。

**第 2 步：拉取模型**

```bash
# 推荐：Qwen2.5（Function Calling 支持最好）
ollama pull qwen2.5:7b

# 备选：Llama 3.1
ollama pull llama3.1:8b
```

**第 3 步：启动 Git Agent**

```bash
go run main.go --api-key ollama --base-url http://localhost:11434/v1 --model qwen2.5:7b
```

或使用环境变量：

```bash
export GIT_AGENT_API_KEY=ollama
export GIT_AGENT_BASE_URL=http://localhost:11434/v1
export GIT_AGENT_MODEL=qwen2.5:7b
go run main.go
```

> ⚠️ **Ollama 使用注意事项**：
> - `--api-key` 可以填任意非空字符串（如 `ollama`），仅用于启用 LLM 模式。Ollama 不需要真实认证。
> - **Function Calling 支持与模型相关**：`qwen2.5` 支持最好，`llama3.1` 有基础支持。不支持 Function Calling 的模型（如 `mistral`、`gemma`）可能导致工具调用失败。
> - 确保 Ollama 服务已运行（`ollama serve`）再启动 Git Agent。
> - 本地模型比云端 API 慢，响应时间取决于您的硬件配置，可能需要几秒钟。

#### 启动成功标志

启动后会看到欢迎界面：

```
╔══════════════════════════════════════════╗
║     🤖 Git Agent - 文件版本管理助手      ║
║         让版本控制像保存一样简单          ║
╚══════════════════════════════════════════╝

🧠 LLM 模式已启用（deepseek-chat @ api.deepseek.com）
   您可以直接用自然语言对话，Agent 会智能理解您的需求。

💡 用自然语言告诉我你想做什么，例如：
   • 保存当前修改
   • 查看修改历史
   • 看看小李改了什么
   • 恢复昨天的版本
   • 输入「帮助」查看更多操作

🧠 您想做什么？ _
```

> 📝 如果是本地模式，开头会显示 `📝 本地模式（关键词匹配）`，使用方式完全一样，只是理解能力稍弱。

## 3. 两种模式怎么选？

| 特性 | 📝 本地模式 | 🧠 LLM 模式 |
|------|------------|-------------|
| **是否需要 API Key** | ❌ 不需要 | ✅ 需要 |
| **理解自然语言** | 基础关键词匹配 | 深度语义理解 |
| **说法灵活性** | 需要用常见说法 | 随便怎么说都行 |
| **对话能力** | 无 | 可以多轮对话 |
| **出错处理** | 提示较简单 | 智能解释和引导 |
| **费用** | 免费 | 按 API 用量计费（本地 Ollama 免费） |
| **推荐场景** | 先试用、简单操作 | 日常使用、复杂需求 |

> 💡 **建议**：先用本地模式熟悉基本操作，等觉得不够智能了再配置 LLM 模式。

## 4. 第一次使用：5 分钟上手

下面用一个完整的例子，带您从零开始体验 Git Agent。

### 步骤 1：启动 Git Agent

```bash
cd 你的工作目录
go run main.go
```

### 步骤 2：初始化仓库

首次在一个新文件夹中使用，需要先初始化：

```
🧠 您想做什么？ 初始化仓库

✅ 文档仓库已创建，您可以开始添加文件了
```

> 💡 这一步就像买了一个新文件夹，准备用来存放和管理您的文档。

### 步骤 3：添加一些文件

在 Git Agent 运行的目录中，用您习惯的方式创建或编辑文件。比如用 Word、记事本、Excel 等创建：

- `市场报告.md` — 一份市场分析报告
- `数据.xlsx` — 一份数据表格

### 步骤 4：保存第一个版本

```
🧠 您想做什么？ 保存修改，这是初稿

✅ 已保存为新版本 #a1b2c3d4
  💡 您可能还想：
     • 推送到远程仓库
     • 查看修改历史
```

> 💡 就像给当前所有文件拍了一张"快照"，以后随时可以回到这个状态。

### 步骤 5：修改文件

用您的编辑工具修改 `市场报告.md`，比如添加了一段分析。

### 步骤 7：保存新版本

```

```

### 步骤 8：查看修改历史


## 5. 常用操作详解

### 5.1 保存文件修改

这是最常用的操作。每次编辑完文件后，告诉 Agent 保存即可。

**您可以这样说：**

| 说法 | 效果 |
|------|------|
| `保存修改` | 保存所有修改的文件 |
| `保存修改，更新了第三章` | 保存所有修改，描述为"更新了第三章" |
| `保存一下` | 保存所有修改 |
| `提交` | 保存所有修改 |
| `暂存` | 保存所有修改 |

**完整示例：**

```
🧠 您想做什么？ 保存修改，完成了市场分析报告的初稿

✅ 已保存为新版本 #c3d4e5f6
  💡 您可能还想：
     • 推送到远程仓库
     • 查看修改历史
```

**只保存指定文件：**

```
🧠 您想做什么？ 保存 市场报告.md 的修改，调整了数据部分

✅ 已保存为新版本 #d4e5f6a7
```

> ⚠️ **注意**：如果没有任何修改，会提示"没有需要保存的修改"。请先编辑文件后再保存。

**您可以这样说：**

| `查看修改记录` | 同上 |
| `看看谁改的` | 显示所有修改记录（含修改人） |
| `历史` | 同上 |

**完整示例：**

```
🧠 您想做什么？ 查看历史

📋 共找到 5 条历史记录：

     • 恢复某个版本

```

| `当前状态` | 同上 |

```
🧠 您想做什么？ 查看状态

  已暂存修改：2 个文件
```

| `看看改了什么` | 同上 |
| `对比差异` | 同上 |
| `有哪些变更` | 同上 |


### 5.5 恢复旧版本

如果文件改坏了或者想回到之前的某个版本，可以用恢复功能。

**您可以这样说：**

| 说法 | 效果 |
|------|------|
| `恢复版本` | 恢复到之前的版本（需要提供版本号） |
| `回到之前的版本` | 同上 |
| `回滚` | 同上 |
| `撤销修改` | 同上 |
| `恢复 a1b2c3d4 这个版本` | 恢复到指定版本 |

> ⚠️ **重要提醒**：恢复整个仓库版本是**不可逆操作**，会丢失当前未保存的修改！建议先保存当前版本，再恢复旧版本。

**完整示例：**

```
🧠 您想做什么？ 恢复版本 e5f6a7b8

✅ 已恢复到指定版本
```

**只恢复某个文件：**

```
🧠 您想做什么？ 恢复 市场报告.md 到 e5f6a7b8 版本

✅ 已恢复到指定版本
```

> 💡 **小技巧**：先用"查看历史"找到版本号，再用"恢复版本"回退。

### 5.6 给版本打标签

给重要的版本起一个好记的名字，比如"终稿"、"v1.0"等，方便以后查找。

**您可以这样说：**

| 说法 | 效果 |
|------|------|

| `标记版本 终稿` | 给当前版本标记为"终稿" |
| `打标签` | 打标签（需提供标签名） |

**完整示例：**

```
🧠 您想做什么？ 打标签 v1.0

✅ 已标记版本「v1.0」
```

> 💡 适合在里程碑节点使用，比如：提交审核前打"审核版"、定稿后打"终稿"。

## 6. 团队协作操作

### 6.1 提交修改给团队

把您的修改推送到远程仓库，让团队成员看到。

**您可以这样说：**

| 说法 | 效果 |
|------|------|
| `提交给团队` | 保存并推送所有修改 |
| `提交审核` | 同上 |
| `申请合并` | 同上 |

**完整示例：**

```
🧠 您想做什么？ 提交给团队，完成了竞争分析

✅ 已同步到远程仓库
```

#### 推送远程仓库的认证方式

首次推送到远程仓库时，可能需要进行身份认证。Git Agent 支持两种协议，认证方式不同：

| 协议 | 地址格式 | 认证方式 | 适合人群 |
|------|---------|---------|---------|
| **HTTPS** | `https://github.com/用户名/仓库名.git` | 用户名 + 访问令牌 | 新手（推荐） |
| **SSH** | `git@github.com:用户名/仓库名.git` | SSH 密钥 | 有经验的用户 |

> 💡 **新建仓库推荐使用 HTTPS + 访问令牌**，对新手最友好。

**HTTPS 认证（推荐新手使用）：**

如果推送时遇到认证失败的提示，Git Agent 会引导您获取访问令牌。流程如下：

1. **Git Agent 检测到认证失败**，会用通俗语言提示您
2. **您去获取访问令牌**（从 GitHub 等平台）
3. **用自然语言告诉 Git Agent 用户名和令牌**
4. **Git Agent 自动用您的凭据重新推送**

**如何获取 GitHub 访问令牌：**

1. 登录 GitHub → 点击右上角头像 → **Settings**
2. 下拉找到 → **Developer settings** → **Personal access tokens** → **Tokens (classic)**
3. 点击 **Generate new token (classic)**
4. 填写描述，勾选 **repo** 权限，点击 **Generate token**
5. **复制令牌**（只显示一次，请妥善保存！）

然后告诉 Git Agent：

```
🧠 您想做什么？ 我的用户名是 jackz-jones，令牌是 ghp_xxxxx

✅ 使用您的凭据推送成功！
```

**从 SSH 切换到 HTTPS：**

如果您的仓库是用 SSH 方式克隆的，但 SSH 密钥配置有问题，可以切换到 HTTPS：

```
🧠 您想做什么？ 推送到远程，使用 HTTPS 地址 https://github.com/jackz-jones/my-project.git，用户名 jackz-jones，令牌 ghp_xxxxx

✅ 远程地址已切换为 HTTPS 并推送成功！
```

**SSH 认证（适合有经验的用户）：**

Git Agent 会按以下顺序自动尝试使用您的 SSH 密钥：

1. **SSH 配置中的 IdentityFile** — 如果您在 `~/.ssh/config` 中为主机配置了 `IdentityFile`
2. **SSH Agent** — 如果您已将密钥加载到 ssh-agent（`ssh-add ~/.ssh/id_rsa`）
3. **默认密钥文件** — `~/.ssh/id_ed25519`、`~/.ssh/id_rsa`、`~/.ssh/id_ecdsa`

如果所有 SSH 认证方式都失败，Git Agent 会建议您切换到 HTTPS 方式（对大多数用户更简单）。

**通过环境变量设置 HTTPS 凭据：**

您也可以通过环境变量预设凭据，避免每次都输入：

```bash
export GIT_HTTP_USERNAME=你的用户名
export GIT_HTTP_PASSWORD=ghp_你的令牌
go run main.go
```

> ⚠️ **前提**：需要先配置远程仓库地址。如果是新仓库，还需要先"推送"一次。

## 7. 工作副本（分支）操作

### 7.1 什么是工作副本？

> 🏢 **通俗理解**：工作副本就像办公室里的"独立工位"。
>
> 比如您和同事同时修改一份报告，如果直接在同一个文件上改，容易互相干扰。工作副本就是给每个人一个独立的修改空间，等各自改完了，再合并到一起。

**常见使用场景：**
- 新开一个方案，不影响主版本 → 创建工作副本
- 试用一个新想法，不确定要不要保留 → 创建工作副本
- 多人同时修改不同部分 → 各自创建工作副本

---

| `开个分支 实验方案` | 创建名为"实验方案"的工作副本 |

## 8. 冲突处理

### 8.1 什么是冲突？

> ⚡ **通俗理解**：冲突就像两个人同时修改了同一份文件的同一个地方。
>
> 比如您把报告第三段改成了"A 方案"，同事把同一段改成了"B 方案"，Git Agent 不知道该听谁的，就需要您来决定。

**冲突产生的典型场景：**
1. 您和同事同时修改了同一个文件的同一个位置
2. 您修改了一个文件，同事删除了同一个文件
3. 您和同事都新增了同名文件

---

### 8.2 检测冲突

主动检查是否有冲突。

**您可以这样说：**

| 说法 | 效果 |
|------|------|
| `检测冲突` | 扫描是否有冲突 |
| `有没有冲突` | 同上 |
| `冲突` | 同上 |

**完整示例（无冲突）：**

```
🧠 您想做什么？ 有没有冲突

📋 冲突检测结果：[] 没有发现冲突
```

**完整示例（有冲突）：**

```
🧠 您想做什么？ 有没有冲突

📋 冲突检测结果：
  📄 报告.md：您和同事都修改了同一位置
```

---

### 8.3 解决冲突

冲突有三种解决策略：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **ours** | 保留我的修改 | 确认自己的版本更准确 |
| **theirs** | 采用对方的修改 | 同事的修改更完善 |
| **merge** | 自动合并双方修改 | 双方修改不重叠，可以合并 |

**您可以这样说：**

| 说法 | 效果 |
|------|------|
| `解决冲突，用 ours` | 冲突文件保留我的版本 |
| `解决冲突，用 theirs` | 冲突文件采用对方版本 |
| `解决冲突，用 merge` | 冲突文件自动合并 |

**完整示例（LLM 模式）：**

```
🧠 您想做什么？ 拉取最新修改

⚠️ 发现 1 处冲突需要处理：
  📄 报告.md：您和同事都修改了同一位置
  💡 建议：冲突区域简单，建议自动合并

🧠 您想做什么？ 解决冲突，用 merge 策略

✅ 冲突已解决！
  📝 报告.md：已自动合并双方修改
  💡 您可能还想：
     • 保存合并结果
     • 提交给团队
```

**完整示例（本地模式）：**

```
📝 您想做什么？ 解决冲突 报告.md 用 merge

✅ 已自动合并双方修改
```

> 💡 **建议**：如果不确定选哪个策略，先用 `merge` 试试。如果自动合并不理想，再手动编辑文件解决。

## 10. 常见问题

### Q1：启动后提示"仓库尚未初始化"怎么办？

**答**：在 Git Agent 中输入 `初始化仓库` 即可。这会在当前目录创建一个文档仓库。

```
🧠 您想做什么？ 初始化仓库

✅ 文档仓库已创建，您可以开始添加文件了
```

---

### Q2：保存时提示"没有需要保存的修改"？

**答**：这说明您的文件自上次保存后没有发生变化。请检查：
1. 您是否真的编辑并保存了文件（有些编辑器需要按 Ctrl+S 保存）
2. 您编辑的文件是否在 Git Agent 运行的目录中

---

### Q3：本地模式和 LLM 模式的说法有区别吗？

**答**：本地模式需要使用**常见的说法**（如"保存修改"、"查看历史"），LLM 模式则更灵活。对比：

| 您想说的话 | 本地模式 | LLM 模式 |
|-----------|---------|----------|
| `保存修改` | ✅ 能识别 | ✅ 能识别 |
| `帮我把改过的东西存一下` | ❌ 可能识别不了 | ✅ 能理解 |
| `我想回到昨天那个版本` | ❌ 无法处理 | ✅ 能理解 |
| `刚才改的报告我不想要了，换回去` | ❌ 无法处理 | ✅ 能理解 |

---

### Q4：恢复版本后还能回到最新版吗？

**答**：可以！恢复版本并不会删除历史记录，您随时可以再次恢复到任何版本。但**恢复前请先保存当前修改**，否则未保存的内容会丢失。

---

### Q5：LLM 模式下 Token 用量是什么意思？

**答**：Token 是大模型计算文字的单位，类似"字数"。每次和 LLM 对话都会消耗 Token，输出中显示的用量让您了解本次操作的费用。例如：

```
  🔋 Token 用量：256 (prompt: 180, completion: 76)
```

- `prompt`：发给大模型的文字量
- `completion`：大模型回复的文字量
- 费用 = 总 Token 数 × 单价（不同模型单价不同）

---

### Q6：如何设置我的名字和邮箱？

**答**：通过环境变量设置，在启动 Git Agent 前输入：

```bash
export GIT_AGENT_USER=张三
export GIT_AGENT_EMAIL=zhangsan@company.com
go run main.go
```

这样保存版本时，修改人就会显示为"张三"。

---

### Q7：如何使用本地 Ollama 模型？

**答**：先确保已安装并启动 Ollama，然后拉取模型并启动 Git Agent：

```bash
# 1. 启动 Ollama 服务
ollama serve

# 2. 拉取模型（推荐 qwen2.5，Function Calling 支持最好）
ollama pull qwen2.5:7b

# 3. 启动 Git Agent
go run main.go --api-key ollama --base-url http://localhost:11434/v1 --model qwen2.5:7b
```

> ⚠️ `--api-key` 必须填非空值才能启用 LLM 模式，但 Ollama 不需要真实密钥，填 `ollama` 即可。

---

### Q8：Ollama 模型不调用工具 / Function Calling 失败

**答**：并非所有 Ollama 模型都同样支持 Function Calling。请尝试以下步骤：

1. **换用支持更好的模型** — `qwen2.5:7b` 或 `qwen2.5:14b` 的支持最好
2. **更新 Ollama** — 旧版本的 OpenAI 兼容性可能不完整：`ollama update`
3. **确认模型支持工具调用** — 部分模型（如 `mistral`、`gemma`、`phi`）的 Function Calling 支持有限或完全不支持

如果 Function Calling 仍然失败，可以去掉 `--api-key` 参数回退到本地模式使用。

在交互模式中输入 `帮助` 或 `help`：

```

  • "查看历史" - 查看修改记录
  • "恢复版本" - 回到之前的版本
  • "查看差异" - 对比修改内容
```

### Q10：推送时提示认证失败怎么办？

**答**：这说明远程仓库需要身份验证。Git Agent 会引导您完成认证。分两种情况：

**情况 A：远程仓库使用 HTTPS（地址以 `https://` 开头）**

Git Agent 会提示您提供用户名和访问令牌，用自然语言告诉它即可：

```
🧠 您想做什么？ 推送，用户名 jackz-jones，令牌 ghp_xxxxx
```

获取 GitHub 访问令牌的步骤：登录 → 右上角头像 → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token → 勾选 repo 权限 → 生成 → 复制令牌。

**情况 B：远程仓库使用 SSH（地址以 `git@` 开头）**

如果 SSH 认证失败，Git Agent 会建议您切换到 HTTPS 方式，更简单：

```
🧠 您想做什么？ 推送，用 HTTPS 地址 https://github.com/用户名/仓库名.git，用户名 jackz-jones，令牌 ghp_xxxxx
```

如果您想修复 SSH，请确保：
1. SSH 密钥已加载：`ssh-add ~/.ssh/id_rsa`
2. `~/.ssh/config` 中配置了正确的 `IdentityFile`

> 💡 **新手建议**：切换到 HTTPS 方式最简单！
