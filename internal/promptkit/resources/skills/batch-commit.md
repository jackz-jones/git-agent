# 分批提交操作技能

当用户要求分多次提交时（如"先提交代码再提交文档"、"先提交文档类修改，再提交代码类修改"），你具备按步骤执行分批提交的能力。

## 执行流程

分批提交需要按以下步骤逐步执行：

1. **理解用户的分组方式**：明确用户想按什么维度拆分提交（按文件类型、按目录、按功能模块等）
2. **确认每一批的范围**：清晰列出每批包含哪些文件，确保不遗漏、不重叠
3. **逐批执行**：
   - 对当前批次：先用 view_diff 查看该批次涉及的修改内容
   - 总结该批次的 commit message（遵循 commit-message 规范）
   - 调用 save_version/submit_change 完成该批提交
   - 告知用户该批已完成
4. **继续下一批**：重复步骤 3，直到所有批次提交完毕

## 示例场景

用户说："先提交文档类修改，再提交代码类修改"

执行步骤：
1. 确认分组：文档类（*.md 等文档文件）vs 代码类（*.go 等源码文件）
2. 第一批（文档类）：view_diff 查看文档修改 → 生成 commit message → save_version → 告知用户"文档类修改已提交"
3. 第二批（代码类）：view_diff 查看代码修改 → 生成 commit message → save_version → 告知用户"代码类修改已提交"

## 关键原则：不理解时就问，不要猜测

分批提交是不可逆操作，如果出现以下情况，**必须先向用户确认，不要自行猜测**：

- 用户说的分组方式不够明确（如"按类型分"，但不清楚具体怎么归类）
- 某个文件不确定应该归属哪一批
- 用户的指令和实际的文件修改情况有出入
- 对任何操作步骤存在疑问

**宁可多问一句，也不要带着模糊的理解去执行 commit。** 你可以向用户提出明确的问题来引导他们给出更清晰的指令，例如：
- "您说的文档类具体包括哪些文件？我看到的修改中有 xxx.md 和 yyy.txt，都是文档类吗？"
- "xxx.go 文件的修改看起来既有重构又有新功能，您希望把它归到哪一批？"
- "我想确认一下，您是想分 2 批还是 3 批提交？"
